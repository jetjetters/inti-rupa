## Arsitektur dan Keputusan Teknis

Sebagai Lead Backend dalam proyek **Inti Rupa**, tanggung jawab utama saya adalah merancang fondasi logika bisnis, pengelolaan database, dan memastikan komunikasi antar-layanan berjalan dengan andal. Transisi dari arsitektur monolitik ke *microservices* menuntut beberapa keputusan teknis penting:

1. **Pemilihan FastAPI:** Keputusan menggunakan FastAPI didasarkan pada dukungan asinkron (`async/await`) secara *native* (ASGI). Karena platform kami sangat bergantung pada I/O-bound operations yang berat (seperti memanggil SDK Google Gemini untuk OCR/Ringkasan dan API HuggingFace untuk generasi gambar), arsitektur asinkron memastikan *event loop* server tidak terblokir selama menunggu respon eksternal. Hal ini meningkatkan efisiensi dan kapasitas penanganan request secara bersamaan (*concurrency*) dibandingkan framework tradisional seperti Flask.
2. **Penerapan Database per Service:** Kami memutuskan untuk memisahkan basis data menjadi `auth_db` (untuk data pengguna) dan `ai_db` (untuk sesi interaksi AI). Keputusan ini bertujuan untuk menjaga otonomi masing-masing layanan. Dengan arsitektur ini, perubahan skema pada satu layanan tidak akan merusak layanan lainnya, serta mencegah timbulnya *anti-pattern shared database*.
3. **Desentralisasi Autentikasi (Stateless JWT):** Untuk menghindari overhead komunikasi jaringan, saya menerapkan verifikasi token JWT secara stateless di sisi `ai-service` menggunakan *shared secret key*. Dengan cara ini, `ai-service` dapat memverifikasi token pengguna secara mandiri tanpa harus selalu melakukan request verifikasi sinkron ke `auth-service` setiap kali ada request API masuk.

## Kesulitan Teknis dan Resolusi

Proses pemecahan layanan menjadi *microservices* membawa kompleksitas tersendiri, terutama terkait ketahanan sistem dan validasi data.

1. **Cascading Failure & Penanganan Downtime:** Salah satu tantangan terbesar adalah bagaimana menjaga `ai-service` tetap berjalan ketika `auth-service` mengalami gangguan. Pada awalnya, jika layanan autentikasi mati, seluruh aplikasi akan mengalami crash karena proses sinkronisasi kuota API terputus. 
   Untuk mengatasinya, saya mengimplementasikan pattern **Circuit Breaker** pada fungsi pemanggilan API kuota (`PUT /auth/users/me/increment-api`). Ketika kegagalan koneksi mencapai batas ambang tertentu, sirkuit akan terbuka (OPEN) dan mengalihkan aplikasi ke mode *degraded*. Pengguna terautentikasi tetap dapat menggunakan AI Generator, dan sistem akan menyajikan statistik sementara (`/stats/degraded`) alih-alih menampilkan error total.
2. **Validasi Input & Hardening Keamanan:** Di fase akhir, saya mendapati adanya risiko serangan jika input dari luar tidak divalidasi dengan ketat. Menggunakan Pydantic v2, saya memperkuat skema input backend. Salah satu langkah krusial adalah membatasi panjang input sandi hingga maksimal 128 karakter pada model pendaftaran. Langkah ini penting untuk mencegah potensi serangan *Denial of Service* (DoS) berbasis CPU exhausting saat melakukan proses hashing menggunakan bcrypt pada string yang sangat panjang.

## Pelajaran Berharga

Membangun bagian backend untuk aplikasi terdistribusi ini memberikan beberapa pelajaran penting yang merubah cara pandang saya dalam membangun sebuah perangkat lunak:

1. **Pentingnya Ketahanan Jaringan (Resilience):** Di sistem monolit, kita sering mengabaikan kemungkinan fungsi internal gagal terpanggil. Namun di dunia *microservices*, kegagalan jaringan antar-layanan adalah keniscayaan. Mendesain sistem yang dapat melakukan *graceful degradation* jauh lebih penting daripada berasumsi jaringan akan selalu aman.
2. **Kolaborasi Lintas Peran:** Kesuksesan integrasi backend sangat bergantung pada kejelasan kontrak API. Dengan memperbarui dokumentasi API Contract secara berkala dan menyediakan Swagger UI interaktif bagi tim Frontend (Incha) dan DevOps (Jetro), proses integrasi rute API Gateway dan penanganan CORS menjadi jauh lebih cepat dan minim miskomunikasi.
3. **Penggunaan AI Secara Bijak:** Selama pengerjaan proyek, AI generatif sangat membantu mempercepat penulisan kode awal dan struktur pengujian. Namun, logika arsitektur tingkat tinggi—terutama penanganan *race condition* pada database, integrasi sirkuit, dan audit kualitas kode (*docstrings*)—tetap memerlukan pemahaman mendalam dan keputusan sadar dari diri saya sendiri sebagai pengembang.
